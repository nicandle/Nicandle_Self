# 游戏玩法原型开发清单 (Gameplay Prototypes Checklist)

本文档汇总了两种不同玩法原型的 MVP 开发需求清单。

---

## 1. 原始宝可梦战斗 (Primitive Pokemon Battle)

**核心体验**: 回合制策略 (Turn-based Strategy) + 属性克制 (Type Matchups)

| 序号 | 任务名称 | 所属阶段 | 说明 | MVP实现方式 |
| :--- | :--- | :--- | :--- | :--- |
| 1 | **基础数据结构** | 🏗️ 基础架构 | 定义宝可梦(Stats)与招式(Moves)的数据模型，包含 HP/Atk/Def/Spe 等基础数值。 | 使用 JSON 或 脚本对象 (ScriptableObject) 存储最基础的 ID/Name/Type/Stats，暂不处理个体值/努力值。 |
| 2 | **战斗状态机** | ⚙️ 核心战斗 | 实现 `回合开始` -> `等待输入` -> `行动结算` -> `回合结束` 的完整循环逻辑。 | 简单的状态枚举 (TurnStart, PlayerInput, Execute, End) 串联流程。 |
| 3 | **速度排序逻辑** | ⚙️ 核心战斗 | 根据双方的速度值 (Speed) 决定谁先出手。 | 简单的 `if (p1.spe >= p2.spe)` 判断，暂不处理同速乱数 (Speed Tie) 和优先度。 |
| 4 | **基础伤害公式** | ⚙️ 核心战斗 | 实现 攻击/防御 的数值对抗计算，得出最终扣血量。 | 简化公式：`(Atk * Power / Def) * TypeMod`，暂略过等级修正和随机浮动。 |
| 5 | **属性克制表** | ⚙️ 核心战斗 | 实现属性克制倍率查询 (2x, 1x, 0.5x, 0x)。 | 硬编码 3-4 个核心属性 (水/火/草/电) 的克制关系矩阵，跑通倍率计算即可。 |
| 6 | **招式执行系统** | ⚙️ 核心战斗 | 处理技能的释放、PP扣除、HP更新逻辑。 | 仅实现“直接伤害”类技能 (Physical/Special)，暂不做变化类 (Status) 和异常状态。 |
| 7 | **战斗主界面 (HUD)** | 🎨 UI与表现 | 显示双方宝可梦的形象 (Front/Back Sprite)、名称、等级、HP条。 | 使用静态图片占位 + 标准血条组件 (Slider) + 文本标签。 |
| 8 | **指令交互菜单** | 🎨 UI与表现 | 玩家操作界面：攻击(Fight)、背包(Bag)、宝可梦(Pkmn)、逃跑(Run)。 | 点击“Fight”直接列出 4 个技能按钮，其他功能按钮暂时置灰不可用。 |
| 9 | **战斗日志系统** | 🎨 UI与表现 | 文字播报战斗过程 (e.g., "Pikachu used Thunderbolt!", "It's super effective!")。 | 底部固定文本框，每一步操作刷新显示一行文本。 |
| 10 | **胜负结算** | 🏁 核心流程 | 判定 HP <= 0 时的濒死状态 (Fainted) 及战斗结束逻辑。 | 一方 HP 归零后，直接弹出“You Win / You Lose”文本框并重置场景。 |

---

## 2. 派对乱斗：夺宝大作战 (Party Brawler: Treasure Snatch)

**核心玩法**: 物理乱斗 + 搬运抢分 + **元素环境交互**  
**上线标准**: 具备完整的循环、UI反馈与联机稳定性，可供玩家进行 3-5 分钟的完整对局。

| 序号 | 任务名称 | 所属阶段 | 详细说明 | 上线级实现标准 (Alpha Test) |
| :--- | :--- | :--- | :--- | :--- |
| 1 | **物理角色控制器 (Pro)** | 🏗️ 基础架构 | 稳定的移动与跳跃，能够承受搬运重物时的重心偏移。 | 调优 Active Ragdoll 参数，确保角色在举起 1.5 倍自身体积物体时不会穿模或飞天。 |
| 2 | **网络物品同步 (Net)** | 🏗️ 基础架构 | 确保所有玩家看到的“宝物”位置一致，且抢夺瞬间归属权判定准确。 | 使用 Server-Authoritative (服务器授权) 逻辑。当两个玩家同时抓取时，服务器判定先后顺序，避免“空气拔河”。 |
| 3 | **宝物生成系统** | ⚙️ 核心机制 | 在场地随机位置生成不同分值的物品（金币=1分，钻石=3分，炸弹=-1分）。 | 编写 `ItemSpawner`，支持权重配置。物品需有物理重量（Weight），分值越高越重，移动越慢。 |
| 4 | **抓取与搬运机制** | ⚙️ 核心机制 | 核心操作：双手吸附物体 -> 举起/拖拽 -> 投掷/松手。 | 射线检测 + FixedJoint 连接。需实现“举过头顶”的 IK 动作，以便于搬运时跨越障碍。 |
| 5 | **战斗与掉落 (Disarm)** | ⚙️ 核心机制 | 攻击不仅是为了击晕，更是为了迫使对方“掉落”宝物。 | 当角色受到冲击力 > 阈值（或眩晕）时，强制断开手部关节连接 (`Joint.Break()`)，宝物受惯性飞出。 |
| 6 | **元素状态体系 (Elem)** | 🧪 **元素机制** | 引入 **水/火/雷** 三种基础元素对角色的物理影响。 | **火**: 烫手 (强制松开抓取物) + 乱跑 (输入反向)。<br>**水**: 打滑 (地面摩擦力极低) + 冲力 (被水柱击飞)。<br>**雷**: 麻痹 (身体僵直/倒地 2s) + 传导 (范围触电)。 |
| 7 | **元素交互逻辑 (React)** | 🧪 **元素机制** | 元素与环境/道具的组合反应，创造意外性。 | **导电**: 雷属性道具扔进水池 -> 水池内所有玩家触电。<br>**灭火**: 拿着着火的道具跳进水里 -> 消除火状态。<br>**蒸发**: 火遇水产生大量烟雾 (阻挡视野)。 |
| 8 | **场地与陷阱** | 🗺️ 关卡设计 | 场地本身提供干扰，结合元素机制。 | 地图需包含：<br>1. **水域**: 减速/导电区域。<br>2. **喷火口**: 定时喷火，点燃经过的玩家。<br>3. **绝缘高台**: 安全的得分区。 |
| 9 | **得分区逻辑 (Zone)** | ⚙️ 核心机制 | 玩家必须将物品搬运到己方（或公共）得分区才能转化为分数。 | 场景设置 Trigger 区域。物品进入区域 -> 销毁物品 -> 播放特效 -> 对应玩家/队伍加分 -> UI跳字。 |
| 10 | **HUD 与 计分板** | 🎨 UI与表现 | 实时显示当前战况，营造紧张感。 | 顶部常驻倒计时 (3:00)。屏幕角落显示实时排名 (Rank) 和当前分数。头顶显示玩家 ID 及**当前元素状态图标**。 |
| 11 | **完整游戏循环** | 🏁 流程控制 | 匹配 -> 倒计时 -> 战斗开始 -> 倒计时结束 -> 结算排名 -> 返回大厅。 | 实现 `GameManager` 状态机。结算界面需展示 MVP（得分最多）和“捣蛋鬼”（击晕最多）。 |
| 12 | **音效与反馈 (Juice)** | 🎨 表现反馈 | 强化物理碰撞的滑稽感和得分的爽快感。 | 必须包含：挥拳空挥音效、拳肉碰撞音效、物品落地音效、得分时的“叮”声、**元素触发音效 (滋滋声/燃烧声)**。 |
